## 一、项目设计的目标

通过本设计环节，学生掌握了SaaS多租户架构的设计与实现方法、前后端分离全栈应用开发技术、关系型数据库建模与优化策略、大语言模型工程化集成与降级容错机制，具备了从需求分析、架构设计、编码实现到测试部署的完整软件工程实践能力，能够独立完成企业级Web应用系统的全流程开发。

## 二、拟解决的工程问题

当前健身行业面临以下工程问题：

1. **数据孤岛问题**：传统健身管理中，用户训练数据、健康指标、饮食信息分散在不同应用中，缺乏统一的数据管理平台，无法实现跨维度的联动分析与趋势追踪。

2. **服务同质化问题**：现有健身应用普遍采用"千人一面"的固定内容推送模式，无法根据用户的实际训练历史和身体指标生成个性化训练建议，用户留存率低。

3. **信息化门槛高**：中小型健身房缺乏低成本的会员管理、课程排期、数据统计等信息化工具，仍大量依赖纸质记录或功能单一的本地软件，运营效率低下。

4. **AI能力集成困难**：大语言模型在健身领域的应用尚处于起步阶段，现有方案普遍缺乏用户真实数据驱动的上下文构建和多供应商适配降级机制，难以将AI能力稳定集成到业务系统中。

5. **部署运维复杂**：传统健身软件部署方式对环境依赖强，缺乏容器化与自动化部署方案，中小型客户难以快速上线和独立运维。

本项目通过设计并实现一个基于SaaS模式的智慧健身管理系统，综合运用多租户数据隔离、AI上下文注入与降级容错、游戏化激励、订阅计费、容器化部署等技术手段，系统性地解决上述工程问题。

## 三、设计内容

### 1、计算

（1）**BMI身体质量指数计算**：根据用户录入的体重与身高数据，自动计算BMI值（BMI = 体重(kg) / 身高(m)²），并按世界卫生组织标准划分为偏瘦（<18.5）、正常（18.5-24.9）、偏胖（25-29.9）、肥胖（≥30）四个等级，为用户健康评估提供量化依据。

（2）**训练负荷强度计算**：基于训练日志中的组数、重量、次数数据，计算每次训练的总训练量（总训练量 = Σ(组数 × 重量 × 次数)）和平均训练强度，为训练计划调整提供数据支撑。

（3）**健康数据统计计算**：对用户健康指标进行时间序列聚合计算，包括平均值、最大值、最小值、变化趋势（差值与变化率），服务于健康趋势展示与AI上下文构建。

（4）**连续打卡天数计算**：基于窗口函数的连续日期段识别算法，先按日期去重，再通过日期与行号的差值分组，差值相同的日期属于同一连续段，取最长段长度即为当前连续打卡天数，用于成就检测和用户激励。

（5）**订阅计费计算**：根据套餐类型（免费/月度29.9元/年度299元）和订阅起止日期，计算订阅金额、到期日期，到期自动检测并更新订阅状态。

### 2、性能验算

（1）**数据库查询性能验算**：针对用户登录（按email/username查询）、训练日志列表（按user_id + workout_date联合查询）、健康数据趋势（按user_id + record_date范围查询）等高频查询场景，设计B-tree索引并进行查询计划分析（EXPLAIN），确保API接口响应时间不超过500ms。

（2）**连接池容量验算**：PostgreSQL连接池参数配置（max: 20, idleTimeout: 30s, connectTimeout: 2s），验证在100并发用户场景下连接池是否满足需求，避免连接耗尽导致服务不可用。

（3）**AI接口性能验算**：验证AI健身助手在不同供应商（OpenAI/DeepSeek/Ollama/Anthropic）下的响应时间与Token消耗，设置max_tokens=1000限制单次对话开销，确保AI调用不阻塞核心业务流程。

（4）**前端首屏加载性能验算**：验证Vite构建产物的资源体积、路由懒加载效果、Ant Design按需加载策略，确保首屏加载时间在可接受范围内。

### 3、计算机优化设计

（1）**SaaS多租户架构优化**：采用共享数据库共享Schema方案，通过gym_id外键实现行级逻辑隔离，在数据隔离性与系统成本之间取得平衡；设计四级角色权限体系（user/coach/gym_admin/admin），前后端双重权限控制确保数据安全。

（2）**AI多模型适配架构优化**：设计"统一接口+策略路由"模式，通过API_ENDPOINTS配置映射屏蔽供应商差异（OpenAI兼容格式 vs Anthropic独立格式），新增供应商仅需添加端点配置无需修改核心逻辑；实现降级容错机制，AI不可达时自动回退至领域知识模板（FALLBACK_RESPONSES），保障服务连续性。

（3）**Prompt工程优化**：设计分层提示词体系，包含系统角色约束（SYSTEM_PROMPT）、三类专项提示词（训练建议/营养指导/计划推荐）和降级预设回复；通过buildUserContext函数从训练日志、健康指标和统计数据中提取个性化信息注入Prompt，实现数据驱动的个性化建议。

（4）**事件驱动成就检测优化**：训练完成后自动触发checkAndUnlockAchievements函数，在数据库事务内完成进度计算与解锁写入，避免异步一致性问题；17项成就覆盖训练次数、连续打卡、累计时长、消耗热量四个维度，增强即时反馈。

（5）**数据库索引优化**：针对高频查询场景建立复合索引（workout_logs.user_id + workout_date、health_data.user_id + record_date），对email和username建立唯一索引加速登录查询。

（6）**前端状态管理优化**：采用Zustand轻量级状态管理方案替代Redux，减少模板代码和性能开销；Axios请求拦截器统一Token注入与401自动跳转登录，路由守卫实现认证状态控制。

### 4、制图

（1）**系统架构图**：绘制三层架构图（表现层/业务逻辑层/数据访问层），展示React SPA前端、Express RESTful API后端和PostgreSQL数据库之间的通信流程，标注JWT认证、CORS跨域、Docker容器化等关键节点。

（2）**数据库ER图**：绘制12张核心表（users、gyms、gym_members、fitness_plans、workout_logs、workout_sets、health_data、exercises、subscriptions、achievements、user_achievements、ai_conversations）及其外键关联关系的实体关系图，标注主键、外键和联合唯一约束。

（3）**AI模块流程图**：绘制AI健身助手的完整请求处理流程，包括用户请求→认证鉴权→速率限制→上下文构建（buildUserContext）→多模型策略路由→Prompt组装→LLM调用→降级容错→响应返回的完整链路。

（4）**SaaS多租户数据隔离示意图**：绘制共享数据库共享Schema方案下，通过gym_id外键实现行级逻辑隔离的示意图，展示不同健身房用户的数据访问边界和权限控制逻辑。

（5）**订阅计费状态流转图**：绘制订阅状态的完整生命周期（active→cancelled→expired），展示订阅开通、续费、取消和到期自动检测的状态转换条件与触发事件。

（6）**成就检测时序图**：绘制训练完成→统计查询→进度比较→解锁写入的时序交互流程，标注数据库事务边界和事件驱动触发点。

## 设计进度安排

1. **第1-2周**：需求分析与系统设计——完成健身行业需求调研、功能与非功能需求分析、系统三层架构设计、12张核心数据表ER图设计、技术选型论证（React 19 + Express 5 + PostgreSQL 15 + Docker）；

2. **第3-4周**：基础框架搭建——搭建前后端项目脚手架（Vite + TypeScript），实现用户认证模块（注册/登录/JWT鉴权/bcrypt加密），完成PostgreSQL数据库建表与连接池配置，实现Zod参数校验体系和统一错误处理中间件；

3. **第5-7周**：核心业务模块开发——实现健身计划管理（模板选用/自定义创建/多维度筛选）、训练日志与动作库（训练组数明细原子写入/动作库CRUD/模糊搜索）、健康数据追踪（7项生理指标录入/BMI自动计算/趋势统计）；

4. **第8-9周**：SaaS多租户与商业模块开发——实现健身房管理（场馆CRUD/会员三级体系basic-premium-vip/会员状态生命周期管理active-expired-suspended）、成就勋章系统（17项成就/事件驱动检测/进度条展示）、订阅计费模块（三级套餐/到期自动检测/事务保证）；

5. **第10-11周**：AI集成与安全加固——实现AI健身助手（多模型策略路由/用户上下文构建/降级容错/速率限制每用户每日20次），完善安全防护（CORS白名单/文件上传限制5MB+类型白名单/AI Prompt注入防护）；

6. **第12-13周**：测试与优化——编写后端API集成测试（Jest + Supertest）和前端组件测试（Vitest + React Testing Library），进行数据库索引优化与查询性能验算，安全测试（401未认证拦截/403越权拒绝/400参数校验/SQL注入防护）；

7. **第14-15周**：部署与文档——完成Docker容器化部署（postgres + backend + frontend三服务编排/健康检查/数据持久化），编写Swagger API文档（34个接口），撰写论文与技术文档，绘制系统架构图、ER图、流程图等；

8. **第16周**：答辩准备——整理项目成果（代码仓库+部署系统+文档），准备答辩PPT和系统演示。
